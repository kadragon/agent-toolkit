# Known Gotchas

- **Rebase not automatic**: after merging a CI infra fix, Dependabot does NOT auto-rebase blocked PRs — always send `@dependabot rebase` explicitly, then re-list PRs (`gh pr list --author app/dependabot --state open`) until the rebased PR appears before proceeding to Phase 3.
- **Dependabot may replace PRs**: after rebase, Dependabot sometimes closes the stale PR and opens a new one with a different number/scope. Re-list and confirm new PR numbers before any merge — never reuse pre-rebase numbers.
- **Branch protection needs GitHub Pro on private repos**: `enable-automerge.sh` on a free-plan private repo gets `403 Upgrade to GitHub Pro` and reports `protection_action: "unsupported_plan"` (exits 0) — `allow_auto_merge` is on with NO required checks, so `--auto` would fire immediately. Treat as merge-on-manual-confirm.
- **No CI signal = same immediate-merge risk**: `protection_action: "skipped"` (no CI checks found) carries the identical risk — `allow_auto_merge` on, no required checks. Treat as merge-on-manual-confirm too.
- **Same-repo parallel merge causes "Base branch was modified"**: merging multiple ready PRs from the same repo in parallel fails because the first merge advances the base before the second lands. Merge same-repo PRs sequentially; cross-repo PRs can run in parallel.
- **Deferred same-repo PRs go stale too**: a PR held back for later action (major-bump review, CI-fix queue, manual confirm) goes stale the moment any other same-repo PR merges earlier — later merge fails with a conflict, not just "base branch modified". Before merging any held-back PR, check whether a same-repo PR merged since triage; if so, treat as `needs_rebase` and send `@dependabot rebase` first.
- **Major PR excluded from grouping can deadlock its own group**: when `groups:` only covers `minor`/`patch`, a major-version bump (e.g. `@cloudflare/workers-types` 4.x→5.x) opens as a separate PR. If a package in the grouped PR (e.g. `wrangler`) has a peer-dependency range tied to that major version, neither PR installs alone — `npm ci` fails ERESOLVE in both directions until both land together. Symptom: `ci_failed` on both PRs in the same repo, with mirrored "Conflicting peer dependency" errors naming each other's package. Fix: combine both diffs into one branch, `npm install` to regenerate the lockfile, verify against the real CI command, push as one PR, close both originals referencing it (see `actions.md` §3e).
